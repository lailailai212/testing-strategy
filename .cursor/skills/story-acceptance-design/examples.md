# Examples — Story Acceptance Design

## 示例 A：Small 模式 — 创建产品

**输入（摘要）**：登录 → 产品列表 → Create → 填表提交 → 成功进详情；名称为空不可提交。

**判定**：Small（单 Story、单主路径、≤8 条）

**输出**：

```markdown
# Story: 产品管理 - 创建产品

**模式**: Small（单 Story、单主路径、预估 4 条）

## 验收条目

| ID | 验收条目（一句话） | 优先级 | 备注 |
|----|-------------------|--------|------|
| 1 | 用户登录后可进入产品列表并看到可点击的 Create 按钮 | P0 | |
| 2 | 用户填写合法必填项提交后，出现创建成功提示并进入新产品详情页 Basic 标签 | P0 | Toast 文案待产品确认 |
| 3 | 产品名称为空时不能提交创建 | P0 | |
| 4 | 创建失败时产品列表不增加该产品 | P1 | QA 扩展 |

## 待确认

- [ ] 登录成功后是否自动 redirect 到产品列表，还是需手动打开 `/product/list`？
- [ ] 成功 Toast 是否同时展示「创建成功」与「Operation success」？
```

## 示例 B：Large 模式 — 仅 Story AC

**输入（摘要）**：创建产品、列表联动、权限等多规则 Story。

**判定**：Large

**输出**（v1.0：不生成测试点表）：

```markdown
# Story: 产品管理 - 创建与列表联动

**模式**: Large

## In Scope
- 创建产品主流程与必填校验
- 创建成功后列表数据更新
- 创建权限控制

## Out of Scope
- 编辑/删除产品

## Story AC
1. 具备权限的用户可完成产品创建并进入新产品详情页。
2. 必填项未填时不能提交创建。
3. 用户创建产品后，产品列表应包含新产品（返回列表时可见）。
4. 无创建权限的用户不能创建产品。

## 待确认
- [ ] 「列表应刷新」是指自动回列表，还是用户返回列表时数据更新？

## 测试点
（v1.0 待产品功能 baseline 确认，暂不生成）
```

> 历史参考：baseline 定稿前的测试点表示例见 git 历史或规范升版文档。

## 示例 C：用户强制 Small

**用户**：「小需求，AC 和测试点合并，只做登录跳转列表。」

**行为**：强制 Small，不输出 Layer 1/2 分离结构。

## 示例 D：用户强制 Large

**用户**：「大需求，先写 AC 给开发 review，再拆测试点。」

**行为**：强制 Large，Story AC ≤8 条；测试点表含 TP-ID 与「AC 覆盖 / QA 扩展」。
