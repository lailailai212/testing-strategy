# Reference — Story Acceptance Design

## Small / Large 自动判定

### Small（默认尝试）

- 1 个 Story / 1 个用户目标
- 预估 ≤ 8 条验收条目
- 1 个入口、1 条主路径
- 角色 ≤ 2，无复杂状态机

### Large（任一命中）

- 跨模块或多条主路径
- 预估 > 8 条 AC 或 > 10 个字段/规则
- 多角色权限矩阵
- 分期交付
- 需区分 AC 覆盖与 QA 扩展审计

输出时写清：`判定为 Large，因为 …`

---

## Story AC 输出形态

统一使用 `## Story AC` + 编号段落，**不用表格**：

```
1. 用户进入 XX 页时，应看到 YY。

2. 用户操作 ZZ 后，应出现 WW。
```

条与条之间空一行；P1 / 待确认项在编号后标注，如 `3.（P1 / 待确认）...`。

## 一句话 AC 质量

### 好

- 用户创建产品成功后，返回产品列表时列表应包含新产品。
- 产品名称为空时不能提交创建。
- 无创建权限的用户看不到 Create 按钮。

### 需改写

| 原文 | 问题 | 改法 |
|------|------|------|
| 创建产品体验更好 | 不可验收 | 写可观察结果 |
| 表单校验正常 | 太泛 | 按字段拆条 |
| 列表应刷新 | 「刷新」歧义 | 备注：含新产品 / 或数据重新加载 |

## 测试点拆分规则

1. **主路径**：每条 AC → 至少 1 个 P0
2. **字段规则**：每个必填/格式/范围 → 1 边界或负向（P1）
3. **权限**：每种角色差异 → 1 条（P0 若阻断）
4. **反馈**：Toast / 跳转 / 留页 → 可合并入主路径或单独 TP（Large）
5. **负向默认**：创建/提交类 AC → 建议 1 条失败不生效（Large 标 QA 扩展）

### 优先级（P0 / P1 / P2 / P3）

完整定义见团队文档 `docs/functional-test-case-design-v1.md` — **用例等级定义**。

| 级别 | 含义 |
|------|------|
| P0 | 主流程、阻断性规则；冒烟 / 发版前必跑 |
| P1 | 边界、重要负向；回归必关注 |
| P2 | 低频组合、待确认场景；专项 / 有余力再测 |
| P3 | 体验细节、极低频边缘；不阻塞发版，按需抽测 |

## Large 模式 AC vs 测试点职责

| 层 | 读者 | 粒度 | v1.0 |
|----|------|------|------|
| Story AC | 开发 | 粗、可勾选 | ✅ 按 skill 输出 |
| 测试点 | QA | 细、可执行 | ⏸ 待**产品功能 baseline**（产品、QA 负责人确认） |

Story AC 不可粗到不可测（避免「功能正常」）。

## Large 测试点（v1.0 暂停）

baseline 定稿前**不生成**测试点表。预期方向：测试点与产品功能 baseline 的模块/字段/规则对齐，再映射到 Story AC。

## 待确认（Open Questions）触发词

出现以下情况必须列待确认：

- 刷新、成功、失败的具体表现未定义
- 登录后 redirect vs 手动导航
- 权限：隐藏按钮 vs 禁用 vs 提交后 403
- 列表 filter/分页下创建后的行为
- 错误文案未给出
