# 功能测试用例设计规范 v1.0

**适用范围**：新需求的测试点设计；根据测试点生成功能测试、E2E 测试用例设计与评审。

**目标**：

- AC 要求一眼可懂，简洁明了；
- 同一份用例兼顾：**人工执行可读**、**AI 逐步执行**。

> 需求与用例设计阶段**不考虑**是否自动化、Playwright 可执行性；自动化在后续单独阶段再评估。

**三层结构**：

```text
Story AC / 验收条目（开发 review）
        ↓
测试点（Large 需求时拆分；Small 可与 AC 合并）
        ↓
功能测试 / E2E 用例（统一步骤格式，可人工 / AI 执行）
```

**用例等级**（P0 / P1 / P2 / P3）贯穿 AC、测试点、功能用例三层。团队落地约定见 [附录 C：落地细节](#附录-c落地细节)。

## 用例等级定义

验收条目、测试点、功能用例共用 **P0 / P1 / P2 / P3** 四级。等级表示**测试优先级与执行范围**，与「AC 覆盖 / QA 扩展」正交——后者表示是否来自 Story 字面，等级表示测完后的重要程度。

### 等级一览

| 等级 | 名称 | 定义 | 典型执行时机 |
|------|------|------|--------------|
| **P0** | 核心 / 冒烟 | Story 主流程与**阻断性**规则；不做则功能不可用或无法验收 | 冒烟、提测准入、发版前必跑 |
| **P1** | 回归 | 常见边界、格式校验、重要负向；不影响主流程上线，但回归必关注 | 全量回归、迭代回归 |
| **P2** | 低频 / 探索 | 特殊组合、低频场景、依赖待确认规则 | 时间允许时、专项测试 |
| **P3** | 可选 / 体验 | 非阻断的体验细节、极低频边缘、已知低优或暂缓项；不测不影响功能验收 | 大版本抽测、空闲补测、客户反馈驱动 |

### 各等级判定标准

#### P0 — 必须测、必须过

满足**任一**即可标 P0：

- Story AC 字面主路径（Large 模式下标 `AC 覆盖` 的 happy path）
- 核心入口可用（如：能进列表、能点 Create）
- 阻断性校验（必填、核心权限——无权限不能操作）
- 核心业务结果（创建成功进详情、支付成功出单等）

**不应标 P0**：纯 UI 细节、低频 format 变体、需产品先确认的边界。

#### P1 — 回归要测

- 字段格式 / 范围边界（如 maxTotalVolume 非法格式）
- 重要负向（创建失败时数据不误增）
- 常见分支（编辑态 vs 新建态，若需求明确）
- Large 模式下多数 `QA 扩展` 默认 P1

#### P2 — 有余力再测

- 特殊列表状态（filter / 分页 / 排序下行为）
- 多条件组合、极端数据
- 需求或文案未完全确认的探索项

#### P3 — 可选、不阻塞发版

- 体验类细节（Toast 展示顺序、按钮 hover 提示、非关键文案一致性）
- 极低频或理论边缘（几乎不会发生的操作路径）
- 产品明确标注低优、或 Out of Scope 边缘的「顺手测一下」
- 已知问题已有 workaround，仅在大版本或专项时复测

**不应标 P3**：任何影响功能正确性、数据正确性、权限安全的项（应 P0–P2）。


## 一、Story AC 怎么写

### 目的

供开发 **5–10 分钟**快速评审，与开发快速拉齐「什么叫做完」，不涉及测试步骤细节。

### 怎么写

**推荐句式**（每条一句话）：

```text
[谁 / 在什么前提下] + [做了什么] + [应出现什么可观察结果]
```

**写作原则**：

| 原则 | 说明 |
|------|------|
| 可验收 | 能判断通过 / 不通过 |
| 可测试 | 含可观察结果（Toast、URL、列表数据、按钮可见性） |
| 一条一事 | 一条只验证一个行为或规则 |
| 简洁 | 给开发 review，不写步骤、不写 locator |
| 消歧 | 「刷新、成功、无权限」等有歧义时，在备注或待确认中定义 |

**Small / Large 策略**：

| 模式 | 何时用 | AC 与测试点关系 |
|------|--------|-----------------|
| **Small** | 单 Story、≤8 条、单主路径 | **合并**：AC = 测试点，一张验收条目表 |
| **Large** | 跨模块、多角色、规则多 | Story AC 短（3–8 条，Comment）；测试点**待产品功能 baseline**（v1.0 不写死） |

Large 命中条件（任一）：跨模块 / >8 条 / 权限矩阵 / 分期交付 / 需区分 AC 覆盖与 QA 扩展。

### 怎么根据需求文档设计验收条目

1. **读需求**：提取角色、入口、主路径、字段规则、权限、成功/失败反馈、Out of Scope。
2. **选模式**：按上表 Small / Large 判定。
3. **写条目**：每条 AC 一句话；Large 时 AC 只写范围契约。
4. **补待确认**：跳转、文案、权限表现等未定义项单独列出。
5. **开发评审 checklist**：
   - [ ] 每条有明确可观察结果
   - [ ] 必填 / 格式 / 权限已覆盖或标待确认
   - [ ] Out of Scope 已对齐（Large）

### Story AC Skills

| Skill | 路径 | 用途 |
|-------|------|------|
| `story-acceptance-design` | 仓库 `.cursor/skills/story-acceptance-design/` | 从 PRD / 口述生成 Small 验收条目表或 Large 的 Story AC + 测试点 |

**触发示例**：

- 「根据需求写验收条目，小需求合并 AC 和测试点」
- 「大需求，先写 Story AC 给开发 review，再拆测试点」

### 例子

#### Small — 创建产品（AC = 测试点）

| ID | 验收条目（一句话） | 优先级 | 备注 |
|----|-------------------|--------|------|
| 1 | 用户登录后可进入产品列表并看到可点击的 Create 按钮 | P0 | |
| 2 | 用户填写合法必填项提交后，出现创建成功提示并进入新产品详情页 Basic 标签 | P0 | Toast 文案待确认 |
| 3 | 产品名称为空时不能提交创建 | P0 | |

#### Large — Story AC（给开发）

1. 具备权限的用户可完成产品创建并进入新产品详情页。
2. 必填项未填时不能提交创建。
3. 用户创建产品后，产品列表应包含新产品（返回列表时可见）。
4. 无创建权限的用户不能创建产品。

## 二、测试点怎么写

> **v1.0 范围说明**：Large 需求的测试点将**基于产品功能 baseline** 展开（字段、拆分规则、与 AC 的映射方式待 **@魏来、@张蒙** 确认）。**本节暂不写死约束与示例**；baseline 定稿后单独升版补充。

### 目的

**Large 需求**时，测试点往往比 AC 多很多：AC 保「范围契约」，测试点保「可测覆盖」。Small 需求不单独写测试点，直接用第一节验收条目表。

### 当前版本约定

| 项 | 约定 |
|------|------|
| Small | 仍按第一节「验收条目表」执行（AC = 测试点） |
| Large — Story AC | 按第一节 Large 示例，写在 Story Comment，产品确认 |
| Large — 测试点 | **待产品功能 baseline**；确认前不在本规范中预设表字段与拆分规则 |
| 负责人 | @魏来、@张蒙 对齐 baseline 细节 |

### 约束点

**TBD**（baseline 定稿后补充），预期方向包括但不限于：

- 与产品功能 baseline 的模块 / 字段 / 规则对齐
- 测试点与 Story AC 的关联方式
- 优先级（P0–P3）分配规则
- `AC 覆盖` / `QA 扩展` 标注方式

### Skills

Large 模式下 **`story-acceptance-design` 仅输出 Story AC + 待确认项**；测试点表在 baseline 定稿后再启用（或另建 skill）。

## 三、根据测试点写测试用例

### 目的

将 AC / 测试点（一句话）展开为**可执行用例**。同一份用例须满足：

| 读者 | 要求 |
|------|------|
| 人工测试 | 步骤清晰、期望可核对 |
| AI 执行 | 每步单一动作、操作对象明确、期望可判定 |

### 如何实现

**统一用例结构**（飞书 / Markdown / 用例库同一模板）：

```markdown
## [用例标题]

| 字段 | 内容 |
|------|------|
| 用例 ID | TC-{模块}-{序号} |
| 优先级 | P0 / P1 / P2 / P3（见 [用例等级定义](#用例等级定义)） |
| 类型 | 功能 / E2E |
| 前置条件 | ... |
| 测试数据 | [DATA: 章节/序号]（见测试数据手册） |

### 步骤与期望

| 步骤 | 操作 | 期望 |
|------|------|------|
| 1 | ... | ... |
| 2 | ... | ... |
```

**步骤与期望的写法要点**：

```text
操作：动词 + [页面/区域] + 控件类型 + 「可见名称」
期望：可观察结果（URL、Toast 文案、页面状态、列表数据）
```

### 约束点

#### 1. 测试数据（飞书文档维护）

**原则**：敏感信息（密码、验证码）不进用例正文；业务数据集中维护在 **《测试数据手册》** 飞书文档。

**手册结构建议**：

```text
《测试数据手册》
├── 0. 使用说明（引用格式、环境说明）
├── 1. 账号与权限
│   ├── 1.1 具备产品创建权限的管理员
│   └── 1.2 无创建权限的普通用户
├── 2. 产品管理
│   ├── 2.1 合法产品名称规则
│   ├── 2.2 非法产品名称样例
│   └── 2.3 合法 maxTotalVolume
└── 3. 环境基线（SIT/UAT base_url 说明，指向 environments.yaml）
```

**数据条目必填字段**：

| 字段 | 说明 |
|------|------|
| 序号 | 章节编号，如 `2.1` |
| 引用键 | 短键，如 `PROD-NAME-VALID` |
| 环境 | SIT / UAT / 通用 |
| 字段说明 | 每个字段含义与合法/非法边界 |
| 示例值 | 非敏感示例；动态值写生成规则 |

**用例中引用格式**（二选一，团队统一用一种）：

- 推荐：`[DATA: 2.1]` 或 `[DATA: 产品管理/2.1 合法产品名称]`
- 完整：`参考《测试数据手册》§2.1`

**步骤示例**：

```text
3. 在产品名称输入框输入产品名称 [DATA: 2.1]
4. 在 maxTotalVolume 输入 [DATA: 2.3]
```

**禁止**：步骤里写死仅本次有效的 magic string，除非该用例 explicitly 测「特定固定数据」并在数据手册有对应条目。

#### 2. 测试用例标题命名

**标题格式**（一眼能懂，不含编号）：

```text
[模块] {场景}-{关键期望}
```

| 段 | 规则 | 示例 |
|----|------|------|
| 模块 | 可选；同一 Story 内可省略 | 产品管理 |
| 场景 | 动宾短语，≤15 字 | 合法提交创建产品 |
| 关键期望 | 可观察结果摘要 | 进入详情Basic页 |

**完整标题示例**：

```text
产品创建-合法提交-进入详情Basic页
产品创建-名称为空-不可提交
登录-验证码登录-进入产品列表
```

**元数据（与标题分开维护）**：

| 字段 | 用途 | 示例 |
|------|------|------|
| 用例 ID | 用例库内唯一编号 | `TC-PROD-001` |
| 优先级 | 单独字段，不写进标题；取值见 [用例等级定义](#用例等级定义) | P0 |

**用例 ID 规则**：`TC-{模块缩写}-{序号}`，Story 内从 001 递增。

**一眼懂的原则**：读标题应知道「测什么场景 + 期望什么结果」。

#### 3. 测试步骤约束

**步骤写法（兼顾人工 / AI）**：

| 规则 | 说明 | 好 | 差 |
|------|------|----|----|
| 一步一动作 | 每步只做一件事 | 点击 Create 按钮 | 点击 Create 并填写表单 |
| 动词开头 | 打开 / 点击 / 输入 / 选择 / 验证 / 等待 | 点击 [产品列表] 工具栏按钮「Create」 | Create 按钮 |
| 操作对象明确 | 页面 + 控件类型 + 可见名称 | 在 [创建产品] 表单字段「Product Name」输入 … | 在第二个输入框输入 |
| 数据外置 | 业务数据引用 DATA | 输入产品名称 [DATA: 2.1] | 写死一次性字符串 |
| 期望可观察 | 对应步骤或汇总期望列 | Toast 展示「创建成功」 | 创建成功 |
| 避免实现细节 | 不写 CSS / xpath / API 路径 | 进入产品详情页 Basic 标签 | `#basic-tab.selected` |
| 步骤数 | 单用例建议 ≤12 步 | — | 20+ 步应拆用例 |

**元素描述模板**：

```text
[页面/区域] + [控件类型] + 「可见名称」
```

**期望结果模板**：

```text
- 页面：URL 含 /product/detail/；或标题为 {产品名}
- 提示：Toast 展示「创建成功」
- 列表：表格中存在列值 = [DATA: 2.1] 的行
- 负向：仍停留在创建页；字段「Product Name」下展示必填错误
```

| Skill | 状态 | 用途 |

| `functional-test-case-design` | 🔜 待建 | 从 ID / TP-ID 展开为用例（步骤 + DATA 引用） |
