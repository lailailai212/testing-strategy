# Story AC 与验收条目指南

## 目的

供开发在 **5–10 分钟**内完成评审：范围是否清晰、是否可验收、是否有遗漏/歧义，并与 QA 在「什么叫做完」上拉齐。

后续功能测试用例从同一套条目 ID 向下展开，避免 AC 与测试点文案重复维护。

**用例等级**（P0 / P1 / P2 / P3）定义见 [functional-test-case-design-v1.md — 用例等级定义](functional-test-case-design-v1.md#用例等级定义)。

---

## 一句话 AC 怎么写

### 推荐句式

```text
[谁/在什么前提下] + [做了什么] + [应出现什么结果]
```

示例：

> 用户创建产品成功后，返回产品列表时列表应包含新产品。

### 写作原则

| 原则 | 说明 |
|------|------|
| 可验收 | 能判断通过 / 不通过，避免「体验更好」「尽量」 |
| 可测试 | 含可观察结果（URL、Toast、列表数据、权限） |
| 一条一事 | 一条只验证一个行为或规则 |
| 范围边界 | 复杂 Story 用 In Scope / Out of Scope |
| 模糊词消歧 | 「刷新、成功、无权限」在备注或待确认中定义 |

### 开发评审 Checklist

- [ ] 每条有明确可观察结果
- [ ] 必填 / 格式 / 权限规则已覆盖或标注待确认
- [ ] 成功与失败反馈已说明（Toast、是否留页、是否跳转）
- [ ] Out of Scope 已标注（Large 模式）

## Small / Large 模式判定

### Small（合并：AC = 测试点）

满足**大部分**：

- 1 个 Story / 1 个用户目标
- 预估 **≤ 8 条**验收条目
- **1 个入口、1 条主路径**
- 角色 ≤ 2 种，无复杂状态机

### Large（分开：Story AC + 测试点）

**命中任一**即切换：

- 多条主路径或跨模块（如 创建 + 审批 + 同步）
- 预估 **> 8 条** AC，或字段 / 规则 **> 10 个**
- 多角色权限矩阵
- 分期交付（P0 上线、P1 后续）
- 需审计「AC 覆盖 vs QA 扩展」

未指定模式时，Agent 按上表自动判定，并在输出中注明采用的模式。

## Small 模式产出

一张**验收条目表**，开发与 QA 共用同一句话：

| 字段 | 说明 |
|------|------|
| ID | 稳定编号，如 1、2、3 |
| 验收条目 | 一句话，可验收 |
| 优先级 | P0 / P1 / P2 / P3（见团队用例等级定义） |
| 备注 | 待确认、QA 扩展说明 |

模板：[templates/small-mode-template.md](templates/small-mode-template.md)

## Large 模式产出

### Layer 1 — Story AC（给开发 review，宜短）

3–8 条一句话，描述**范围契约**，可勾选，不必逐步骤。写在 **Story Comment**，产品确认。

### Layer 2 — 测试点（给 QA，宜细）

> **v1.0 暂不定义**。Large 需求测试点将**基于产品功能 baseline** 操作；表字段、拆分规则待 **@魏来、@张蒙** 确认后补充。定稿前请参考 [functional-test-case-design-v1.md — 二、测试点怎么写](functional-test-case-design-v1.md#二测试点怎么写)。

模板：[templates/large-mode-template.md](templates/large-mode-template.md)（测试点部分待 baseline 后启用）

## 从条目到用例（第三层）

验收条目 / 测试点仍是一句话；**功能测试用例**在后续阶段再展开（见 [functional-test-case-design-v1.md](functional-test-case-design-v1.md)）。需求阶段不考虑自动化选型。

## 测试点拆分口诀

> **v1.0**：以下口诀适用于 **Small（验收条目表）** 及 baseline 定稿后的 Large 测试点；Large 测试点细则 **TBD**。

> 一条 AC → 至少一个 P0 主路径测试点  
> 每个字段规则 → 一个边界或负向点  
> 每个权限差异 → 一个权限点  
> 含「刷新、成功、正常、无权限」→ 补 1 条负向或边界
